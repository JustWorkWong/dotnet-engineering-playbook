# C# / .NET 生产路线图

## 1. 先定目标

这个路线图不是为了把你训练成“会背答案的人”，而是为了把你训练成下面这种工程师：

> 能写业务、能解释底层、能做架构判断、能处理线上问题、能借助 AI 提升交付速度的 `.NET` 工程师。

更直接一点，生产里真正有价值的能力不是“知道多少名词”，而是：

1. 会用：能把功能做出来
2. 会解释：能说明为什么这样设计
3. 会取舍：知道为什么选这个，不选那个
4. 会兜底：知道失败后怎么观测、恢复、回滚

## 2. 总体主线

建议按 6 条主线并行推进，但优先级不同：

1. 运行时与语言底座
2. Web 与应用层骨架
3. 数据与分布式可靠性
4. 可观测性与运维闭环
5. 架构演化与技术选型
6. AI 开发与 AI coding workflow

核心原则很简单：

```text
先把单体系统做稳
再把跨进程链路做稳
再把可观测性补齐
最后再谈大规模抽象和高自动化
```

## 3. 第一阶段：运行时与语言底座

### 3.1 必须真的理解什么

- `async / await` 的状态机本质
- 线程池如何调度
- `Task`、取消、超时、阻塞之间的关系
- GC、对象分配、LOH、Span、池化的基本边界
- `IEnumerable`、`IAsyncEnumerable`、延迟执行和多次枚举成本

### 3.2 为什么这一层最先补

很多“业务问题”本质上不是业务问题，而是底座问题：

- 请求慢，可能是线程池饥饿
- 内存高，可能是分配和对象生命周期失控
- 吞吐差，可能是同步阻塞混进了异步链
- 接口偶发超时，可能是取消没有正确传播

### 3.3 当前更稳的默认做法

- IO 密集路径默认异步
- 公共 API 默认带 `CancellationToken`
- 热路径先减少分配，再谈复杂优化
- 不要把所有对象都扔进 DI 容器

### 3.4 什么时候不要过度优化

- 没有观测数据前，不要盲目上对象池、Span、手写高性能代码
- 没有热路径证据前，不要为了“更快”牺牲可读性

## 4. 第二阶段：Web 与应用层骨架

### 4.1 必修主题

- ASP.NET Core 请求管线
- Middleware、Filter、Endpoint 的边界
- DI 生命周期：`Singleton / Scoped / Transient`
- Options、配置分层、环境隔离
- 统一异常处理、验证、日志、审计
- `BackgroundService`、定时任务、队列消费者的治理

### 4.2 推荐默认架构

对大多数业务系统，先从 **分层清晰的模块化单体** 开始，比一上来微服务更稳。

更推荐的顺序是：

```text
Controller / Endpoint
  -> Application Service
  -> Domain / Business Rules
  -> Repository / Infra
```

重点不是套某个架构名词，而是把责任切清：

- 接口层负责协议
- 应用层负责流程编排
- 领域层负责业务规则
- 基础设施层负责数据库、缓存、MQ、外部系统

### 4.3 为什么不建议先上复杂架构

- `Clean Architecture` 能带来边界，但也会带来额外抽象成本
- `CQRS + MediatR` 在复杂领域里有价值，在简单 CRUD 里容易把链路打散
- 大量“模式化代码”会让排障成本上升

判断标准：

> 如果复杂度还没有出现，就不要为了“将来可能复杂”先把今天写复杂。

## 5. 第三阶段：数据与分布式可靠性

### 5.1 数据访问路线

建议优先级：

1. 先把 SQL、索引、事务、锁、隔离级别讲清楚
2. 再熟练使用 `EF Core`
3. 性能敏感热点再引入 `Dapper` 或 Raw SQL

### 5.2 选型 trade-off

参见：

- [`EF Core vs Dapper vs Raw SQL`](../02-foundations/ef-core-vs-dapper-vs-raw-sql.md)

#### `EF Core`

适合：

- 业务模型清晰
- 开发效率优先
- 团队希望统一数据访问风格

代价：

- 复杂查询和性能问题需要更强 SQL 素养
- 抽象层会掩盖部分数据库行为

#### `Dapper / Raw SQL`

适合：

- 复杂查询多
- 性能可控性要求高
- 团队数据库基础较强

代价：

- 开发成本更高
- 映射和维护负担更重

### 5.3 分布式可靠性

必须补的主题：

- 事务边界
- Outbox / Inbox
- 幂等
- 顺序性
- 重试与补偿
- 死信与人工处理

这里最重要的观念不是“会用 CAP”，而是理解：

> 跨进程以后，系统默认就不是强一致；你要做的是把不一致控制在可解释、可恢复、可补偿的范围内。

### 5.4 为什么 CAP 值得学

参见：

- [`CAP 生产实践`](../03-distributed-systems/dotnet-cap-production-guide.md)
- [`模块化单体 vs 微服务`](../03-distributed-systems/modular-monolith-vs-microservices.md)

因为它把下面这条生产主线打通了：

```text
业务数据提交
  + 可靠发布事件
  + 消费者幂等
  + 重试与补偿
  + 运维可见性
```

如果直接用 MQ 却没补齐这些能力，代码看着更“轻”，系统实际上更脆。

## 6. 第四阶段：可观测性与运维闭环

### 6.1 生产里至少要看见什么

- 请求慢在哪里
- SQL 慢在哪里
- 消息卡在哪里
- 后台任务堵在哪里
- 错误发生在哪个上下文里
- AI 调用花了多少钱、耗了多少时间、结果值不值得

### 6.2 观测分工

```text
Log    -> 为什么坏了
Trace  -> 慢在哪里，链路怎么串
Metric -> 趋势、容量、告警
Event  -> 用户和业务发生了什么
```

### 6.3 推荐默认做法

- 用 OpenTelemetry 做统一采集层
- 关键业务动作补 Event 埋点
- 告警基于 SLI/SLO，不靠人工盯日志
- Trace 与业务埋点、审计记录能互相回查

### 6.4 为什么很多团队“有日志但没观测能力”

因为他们只回答了“报错了什么”，没有回答：

- 谁受影响
- 哪条链路受影响
- 为什么开始退化
- 是否和某次部署、某个依赖、某段消息积压相关

## 7. 第五阶段：架构演化与技术选型

### 7.1 演化顺序比技术名词更重要

更稳的演化路径通常是：

```text
简单单体
  -> 分层清晰的模块化单体
  -> 按边界拆独立服务
  -> 必要时再引入平台化能力
```

### 7.2 常见选型判断

#### 模块化单体 vs 微服务

优先模块化单体，当：

- 团队不大
- 领域边界还在变化
- 运维能力有限
- 交付速度比组织隔离更重要

再考虑微服务，当：

- 团队和领域边界都比较稳定
- 部署节奏明显冲突
- 扩容、隔离、自治收益已经高于治理成本

#### Aspire / Dapr / ABP 要不要上

- `Aspire`：更偏本地编排、开发体验、服务连接和观测接线
- `Dapr`：更偏分布式 building blocks，不是免费架构升级
- `ABP`：更偏企业应用框架，能提速，但会带来框架约束

原则：

> 先问团队缺什么能力，再问这个框架补了什么，不要因为“别人用了”就上。

### 7.3 任何架构都要回答的 5 个问题

1. 业务边界怎么划
2. 状态存在哪里
3. 故障怎么扩散
4. 观测怎么补
5. 回滚和补偿怎么做

## 8. 第六阶段：AI 应用与 AI coding workflow

### 8.1 先把 AI 放回工程语境

对 `.NET` 团队来说，AI 不是孤立赛道，而是三层问题：

1. 如何在业务系统里接入 AI 能力
2. 如何对 Agent / Workflow / RAG 做工程化治理
3. 如何用 `Claude Code / Codex / ChatGPT` 放大研发效率

### 8.2 业务系统里的 AI 选型顺序

```text
Prompt + Structured Output
  -> Tool Calling
  -> RAG
  -> Workflow
  -> Multi-Agent
```

不要反过来。

很多团队一开始就上 Agent，不是因为问题真的需要，而是因为看起来高级。结果通常是：

- 成本高
- 状态难管
- 结果难评估
- 故障难复盘

### 8.3 Claude Code / Codex / ChatGPT 怎么分工更稳

#### Claude Code

更适合：

- 在本地仓库里长时间推进任务
- 强依赖仓库规则、hooks、子代理协作
- 需要把“读代码、改代码、验证”串成闭环

#### Codex

更适合：

- 基于 `AGENTS.md` 做结构化任务推进
- 在明确边界内做实现和验证
- 把任务压成可执行、可检查、可回退的工作单元

#### ChatGPT

更适合：

- 前期方案发散和问题拆解
- 解释原理、比较方案、生成提纲
- 做跨主题对照表、学习卡片、talk track 初稿

不建议让 ChatGPT 单独承担高风险仓库改动闭环，除非外层验证和权限控制已经补齐。

参见：

- [`RAG / Workflow / Agent 分层实践`](../05-ai-coding/rag-workflow-agent-layering.md)
- [`Claude Code / Codex / ChatGPT 协作手册`](../05-ai-coding/claude-code-codex-chatgpt-playbook.md)

### 8.4 AI coding workflow 的稳妥接入顺序

1. 先写清 `AGENTS.md / CLAUDE.md`
2. 再定义最小验证命令
3. 再限制工具与权限边界
4. 再引入多 agent / hooks / MCP
5. 最后才扩大自动化半径

本质上还是老原则：

> 自动化半径永远不能超过你的验证半径。

## 9. 面试在这张图里的位置

面试不是单独一条线，而是所有生产能力的压缩输出层。

每个主题最后都应该能输出三种表达：

1. 1 分钟结论版
2. 3 分钟原理版
3. 10 分钟 trade-off + 案例版

如果一个知识点只能背定义，不能讲失败模式、恢复手段和为什么不选别的，它就还没真正进入生产能力。

## 10. 建议的落地节奏

### 第一轮

- 读路线图
- 读 `.NET` 底座
- 读 CAP
- 读 OpenTelemetry
- 读 AI 协作手册

### 第二轮

- 每个主题补一页自己的 trade-off 笔记
- 每个主题补一个最小 demo
- 每个主题补一个失败案例或排障路径

### 第三轮

- 把生产实践压成面试输出
- 练 1 分钟 / 3 分钟 / 10 分钟表达
- 针对“为什么选这个，不选那个”做对照训练

## 11. 最后一句判断标准

如果一项技术让你：

- 更容易交付
- 更容易解释
- 更容易观测
- 更容易恢复
- 更容易让团队协作

那它大概率是合适的。

如果它只是让架构图看起来更高级，那先别急着上。
