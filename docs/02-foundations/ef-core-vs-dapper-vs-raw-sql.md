# EF Core vs Dapper vs Raw SQL

## 1. 先讲结论

这不是三选一问题，而是三层控制面的选择问题：

- `EF Core`：默认解，优先解决大多数业务开发效率和一致性问题
- `Dapper`：中间解，适合性能敏感但仍想保留对象映射便利的查询场景
- `Raw SQL`：最底层解，适合复杂查询、批处理、运维脚本和必须完全掌控 SQL 的热点路径

对大多数 `.NET` 业务系统，一个更稳的默认答案通常是：

```text
80% 常规业务 CRUD 和事务流程 -> EF Core
15% 复杂查询或热点读路径       -> Dapper
5% 极复杂 SQL / 批处理 / DBA 级优化 -> Raw SQL
```

真正的问题从来不是“哪个更高级”，而是：

1. 你要优化的是开发速度，还是查询可控性
2. 你面对的是普通业务查询，还是高复杂度热点路径
3. 团队有没有足够的 SQL 能力和排障能力

## 2. 这个决策到底在解决什么问题

数据访问技术选型，本质上是在平衡四件事：

1. 代码表达力
2. SQL 可控性
3. 性能与可观测性
4. 团队维护成本

很多团队讨论到最后会跑偏成“ORM 慢不慢”，但更本质的判断其实是：

- 业务规则是否频繁变化
- 查询路径是否简单稳定
- 团队是否能长期维护大量手写 SQL
- 线上问题是否需要快速落到执行计划和数据库行为

## 3. 三者分别解决什么问题

### 3.1 EF Core

`EF Core` 解决的是：

- 把数据库访问纳入统一对象模型
- 把实体关系、变更跟踪、事务和迁移放进同一套开发体验里
- 让应用服务层更容易围绕领域模型组织代码

它最强的地方不是“性能”，而是：

- 开发效率高
- 一致性好
- 团队风格更统一
- 事务和实体状态管理更顺手

### 3.2 Dapper

`Dapper` 解决的是：

- 保留手写 SQL 的可控性
- 同时减少底层 `IDataReader` 映射样板代码

它适合那些已经知道 SQL 应该怎么写，但又不想每次都手搓映射的场景。

### 3.3 Raw SQL

`Raw SQL` 解决的是：

- 你必须精确控制执行语句本身
- 你需要写复杂查询、批处理、窗口函数、CTE、Hint、运维 SQL
- 你不想让抽象层掩盖数据库行为

它最接近数据库真实执行面，也最容易把系统带入“能力上限高，但维护门槛也高”的状态。

## 4. 决策矩阵

| 维度 | EF Core | Dapper | Raw SQL |
| --- | --- | --- | --- |
| 开发效率 | 高 | 中 | 低 |
| SQL 可控性 | 中 | 高 | 最高 |
| 学习成本 | 中 | 中 | 高 |
| 团队统一性 | 高 | 中 | 低 |
| 复杂查询表达 | 中 | 高 | 最高 |
| 热点性能调优 | 中 | 高 | 最高 |
| 迁移与模型管理 | 高 | 低 | 低 |
| 误用风险 | 中 | 中 | 高 |

## 5. 更稳的默认路线

### 5.1 先学 SQL，再决定上层工具

先把这些东西讲清楚，再谈 ORM 或微 ORM：

- 索引
- 执行计划
- 事务
- 锁
- 隔离级别
- 慢查询定位

原因很简单：

> ORM 可以帮你省样板代码，但不能替你理解数据库。

### 5.2 默认从 EF Core 起步

当你面对的是典型业务系统时，先从 `EF Core` 起步通常更稳，因为它天然适合：

- 事务性业务流程
- 实体关系较清晰的领域模型
- 中小团队长期维护
- 应用服务围绕聚合根组织代码

### 5.3 热点再下沉

当你遇到这些信号时，再考虑下沉到 `Dapper` 或 `Raw SQL`：

- 查询已经成为热点
- 生成 SQL 难以解释
- 需要精确控制 join、分页、聚合或批量操作
- 你已经有执行计划和性能数据证明当前路径是瓶颈

这时再做技术下沉，才是“按证据优化”，不是“按偏好优化”。

## 6. EF Core 更适合什么时候

### 6.1 适合场景

- 标准 CRUD
- 有明确聚合边界的业务模型
- 需要事务、变更跟踪、导航关系
- 团队更关心交付效率和一致性
- 迁移希望纳入代码管理

### 6.2 优点

- 实体模型和业务代码更接近
- 迁移、事务、并发控制支持完整
- `LINQ` 对大多数常规查询够用
- 更容易统一仓储、测试和团队写法

### 6.3 代价

- 抽象层可能掩盖真实 SQL
- 复杂查询调优成本会变高
- 误用 `Include`、`ToList`、客户端求值时容易性能失控
- 团队如果不会看生成 SQL，就容易把性能问题推给框架

### 6.4 当前更稳的 EF Core 用法

- 写路径优先让聚合和事务保持清晰
- 读路径默认 `AsNoTracking`
- 热点查询显式看 SQL 和执行计划
- 不把“仓储模式”做成毫无信息量的薄封装
- 复杂查询不要死磕纯 `LINQ`，必要时接受下沉

## 7. Dapper 更适合什么时候

### 7.1 适合场景

- 复杂报表查询
- 高频只读热点
- 查询 SQL 需要人工精确控制
- 团队愿意维护 SQL 但不想手搓映射

### 7.2 优点

- SQL 完全可见
- 对热点查询更容易调优
- 读模型与表结构映射足够轻
- 没有重型跟踪机制干扰

### 7.3 代价

- 迁移、状态管理、关系装配要自己负责
- 大量查询散落后容易失去结构
- 团队 SQL 水平不足时，问题会非常直接暴露

### 7.4 当前更稳的 Dapper 用法

- 优先用在读模型，不要一上来全局替换写路径
- SQL 文件或查询对象要有清晰归属
- 参数化查询、超时、分页、索引命中必须显式治理
- 配合执行计划和指标做热点治理

## 8. Raw SQL 更适合什么时候

### 8.1 适合场景

- 极复杂聚合查询
- 批量更新、归档、对账、运维脚本
- 需要数据库特性，如窗口函数、CTE、Hint、临时表
- DBA 或平台团队要完全掌控语句行为

### 8.2 优点

- 控制力最高
- 能最贴近数据库能力上限
- 适合复杂场景和底层治理任务

### 8.3 代价

- 最容易散落成“SQL 泥团”
- 可测试性、可维护性、代码统一性最难守住
- 对团队数据库基础要求最高

### 8.4 当前更稳的 Raw SQL 用法

- 只用于有明确收益的场景
- SQL 要有模块归属，不要散在控制器和服务里
- 关键语句要配注释、风险说明和验证方式
- 高风险变更要有回滚脚本和观测指标

## 9. 常见误区

### 9.1 “EF Core 一定慢”

不准确。

很多性能问题真正的根因是：

- 查询设计差
- 索引缺失
- 一次拉太多字段或关系
- 没有分清读写模型

### 9.2 “手写 SQL 一定更快”

也不准确。

手写 SQL 只是让你更接近真相，不代表你一定会写得更好。错误的 SQL 同样会很慢，而且更难统一维护。

### 9.3 “技术栈必须统一成一种”

这通常是为了管理方便，但如果走到极端，就会变成错误约束。

更稳的做法是：

> 默认统一，热点例外，但例外必须有证据、有边界、有文档。

## 10. 一个更实用的分层做法

### 写路径

默认优先 `EF Core`：

- 聚合根
- 事务
- 并发控制
- 领域事件前后的业务状态变更

### 读路径

按复杂度分层：

- 简单查询：`EF Core`
- 热点复杂查询：`Dapper`
- 极复杂分析或批处理：`Raw SQL`

这样做的好处是：

- 写路径保持一致性和可维护性
- 读路径保留性能和可控性
- 避免把整个系统绑死在单一工具上

## 11. 什么时候该升级方案

从 `EF Core` 升到 `Dapper` 或 `Raw SQL`，至少应满足这些条件中的两个：

1. 已有真实慢查询证据
2. 生成 SQL 难以稳定解释
3. 查询逻辑明显更适合手写
4. 团队能长期维护这段 SQL
5. 这条路径确实是核心热点

如果只是“感觉 ORM 不优雅”，通常还不够构成升级理由。

## 12. 失败模式

### EF Core 失败模式

- `N+1` 查询
- 滥用导航属性和 `Include`
- 查询对象过大
- 把复杂报表查询硬塞进 `LINQ`

### Dapper 失败模式

- SQL 到处复制
- 参数和映射不统一
- 查询对象和索引演化脱节
- 缺少统一超时和观测

### Raw SQL 失败模式

- 业务代码和 SQL 强耦合
- 大量数据库方言蔓延到应用层
- 运维脚本无回滚、无审计
- 只有作者自己看得懂

## 13. 最后给一个落地建议

如果你在做一个新的 `.NET` 业务系统，更稳的起步方式通常是：

1. 先用 `EF Core` 搭主写路径和大部分业务查询
2. 保持 SQL 可观察，及时看慢查询和执行计划
3. 碰到热点读路径，再局部引入 `Dapper`
4. 只有在复杂度或收益明显成立时，才下沉到 `Raw SQL`

这条路线的本质不是保守，而是：

> 先用高层工具提升交付，再用低层工具解决被证据证明的问题。
