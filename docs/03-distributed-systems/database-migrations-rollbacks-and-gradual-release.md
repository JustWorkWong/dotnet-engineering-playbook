# 数据库迁移 / 回滚 / 灰度发布

## 1. 先讲结论

数据库变更不是“发版顺手改个表”。

更稳的生产原则通常是：

> 所有生产数据库变更都当成独立交付单元；迁移优先前滚，不靠直接回滚；应用发布优先灰度，不赌一次性全量切换。

这三个问题本质上是一条链：

- `迁移`：结构怎么安全变化
- `回滚`：出问题后怎么恢复
- `灰度发布`：如何降低一次切换的爆炸半径

如果这条链没设计好，最常见的事故不是“功能错了”，而是：

- 锁表
- 迁移超时
- 新旧代码不兼容
- 数据回填把主库打满
- 回滚代码成功，但数据库已经回不去

## 2. 这套设计到底在解决什么问题

它本质上在解决四件事：

1. 结构变化如何进入生产
2. 新旧版本如何在同一段时间共存
3. 大数据量变更如何避免长时间锁表
4. 出事后如何用最小代价恢复

很多团队把数据库迁移理解成“ORM 帮我们生成 SQL”，但真正的难点不在工具，而在：

- 数据量
- 锁
- 兼容性
- 发布顺序

## 3. 先把三个概念分开

### 3.1 迁移

迁移回答的是：

- 表结构怎么变
- 索引怎么加
- 列怎么加删改
- 数据怎么回填

### 3.2 回滚

回滚回答的是：

- 如果这次发布失败，怎么恢复服务能力
- 是回代码、回流量，还是再发一个前滚修复

注意：

> 在生产里，数据库更稳的做法通常不是依赖“DOWN migration 回到过去”，而是依赖新的前滚迁移把系统带回安全状态。

### 3.3 灰度发布

灰度回答的是：

- 如何让新版本逐步接流量
- 如何在新旧版本共存时保持兼容
- 如何在发现异常时快速止损

## 4. 更稳的默认原则

### 4.1 每次变更都是 migration

不要直接手工改生产库，再想着“回头补脚本”。

因为一旦没有标准化迁移：

- 环境差异会越来越大
- 演练和回放几乎不可能
- 出问题时没人能说清当前库到底是什么状态

### 4.2 生产迁移优先前滚

很多团队喜欢幻想“有 DOWN 就安全”，但真实生产里更常见的是：

- 数据已经写入新列
- 新旧代码都跑过了
- 部分回填已经完成
- 外部依赖已经开始使用新字段

这时强行回滚数据库，风险往往更大。

所以更稳的原则通常是：

```text
代码可以回旧版本
数据库迁移优先发新的修复迁移
```

### 4.3 DDL 和 DML 分开

不要把下面几件事混成一个迁移：

- 加列
- 建索引
- 全量回填
- 删除旧字段

更稳的顺序通常是：

1. 先扩结构
2. 再双写或兼容
3. 再回填数据
4. 再切读路径
5. 最后再收缩旧结构

### 4.4 所有变更都按大表来设计

就算今天只有 1 万行，也按明天 1000 万行来想。

因为很多迁移在小表上看起来没问题，一上生产就会变成：

- 长事务
- 锁等待
- 日志暴涨
- 复制延迟

## 5. 一个更稳的迁移模型：Expand / Migrate / Contract

这是生产里最值得默认采用的模式。

### 5.1 Expand

先做不会破坏旧代码的扩展：

- 新增可空列
- 新增新表
- 新增索引
- 新增兼容字段

原则是：

> 老版本应用继续跑，也不会出错。

### 5.2 Migrate

让新旧结构短时间共存：

- 应用开始双写
- 后台任务做回填
- 新代码开始兼容读

这个阶段最容易踩坑，因为系统处于“双世界并存”状态。

### 5.3 Contract

确认旧结构已经不再被读取或写入后，再收缩：

- 删除旧列
- 删除旧索引
- 删除旧分支逻辑

很多事故都发生在过早收缩。

## 6. 常见场景的更稳默认解

### 6.1 加列

更稳的默认做法：

- 先加可空列或带安全默认值的列
- 应用先兼容“旧数据没有这个值”
- 回填单独做

不要一上来就在大表上加：

- `NOT NULL` 且无默认值
- 会触发表重写的结构变更

### 6.2 加索引

更稳的默认做法：

- 在大表上优先使用并发/在线方式
- 索引构建和高峰流量错开
- 先验证真实查询收益，再建索引

不要把“多建几个索引”当成万能答案，因为代价也很真实：

- 写放大
- 维护成本
- 迁移时间增长

### 6.3 回填数据

更稳的默认做法：

- 分批
- 可恢复
- 可重复执行
- 有进度和限速

不要一条 SQL 全表扫到底，尤其在主业务表上。

### 6.4 改字段语义

比如：

- `status` 改枚举
- 金额精度变化
- 时间字段时区语义变化

这类变更更危险，因为它不是简单加删列，而是“业务含义变化”。

更稳的默认做法通常是：

- 新字段承载新语义
- 老字段保留兼容窗口
- 明确切换时点
- 逐步迁移调用方

### 6.5 删除字段或表

永远最后做。

因为删除是最难恢复的结构变更之一。

更稳的顺序通常是：

1. 先停写
2. 再停读
3. 观察一段时间
4. 最后删除

## 7. 回滚到底该怎么理解

### 7.1 代码回滚不等于数据库回滚

这是最容易讲错的地方。

很多系统在发布失败时，最先做的是：

- 把应用回滚到上一个版本

这没问题，但前提是：

- 上一个版本仍兼容新结构

所以数据库迁移必须优先支持“旧代码还能活”。

### 7.2 数据库恢复的更稳思路

出问题时，优先顺序通常是：

1. 先停新流量或切回旧代码
2. 再判断数据库结构是否还能兼容旧版本
3. 再决定是前滚修复、人工补偿，还是数据恢复

更稳的默认恢复方式通常不是“直接 DROP 新列”，而是：

- 关掉新逻辑
- 停止写入新语义
- 发一个修复迁移

### 7.3 真正需要数据库恢复的场景

只有在这些高风险场景下，才可能进入真正恢复链：

- 数据被错误覆盖
- 错误删除
- 错误批量更新
- 不可逆结构破坏

这时要依赖的是：

- 备份
- PITR
- 数据修复脚本
- 人工补偿流程

不是单纯指望 migration down。

## 8. 灰度发布的更稳默认解

### 8.1 为什么数据库变更必须和灰度一起想

因为应用和数据库并不是同一时刻一起切换的。

真实发布通常会经历：

- 部分实例先升级
- 部分流量先进入新版本
- 后台任务开始用新逻辑
- 老实例还没下线

所以真正关键的问题是：

> 在新旧版本共存期间，系统是否仍然正确。

### 8.2 更稳的灰度顺序

```text
先上兼容数据库结构
  -> 再发兼容代码
  -> 再小流量启用新逻辑
  -> 观察指标
  -> 再逐步放量
  -> 最后移除旧结构
```

### 8.3 灰度时必须观察什么

- 错误率
- 慢 SQL
- 锁等待
- 数据回填进度
- 数据一致性抽样
- 新旧版本写入差异

### 8.4 功能开关比代码分支更值钱

如果数据库变更关联业务行为切换，更稳的做法通常是：

- 结构先上线
- 行为用 feature flag 控制

这样你能：

- 快速关掉新逻辑
- 不必重新发版才能止损

## 9. `.NET` 里的更稳落地方式

### 9.1 EF Core Migrations

适合：

- 作为结构变更登记和版本管理入口
- 中小团队统一 schema 演化

但要注意：

- 不要把所有大数据量回填都塞进同一个 migration
- 复杂索引和在线变更必要时用手写 SQL

### 9.2 后台回填

适合用：

- `BackgroundService`
- 独立 worker
- 队列驱动任务

前提是必须有：

- 幂等
- 断点续跑
- 限速
- 进度观测

### 9.3 发布编排

更稳的顺序通常是：

1. 先跑 schema migration
2. 再发兼容代码
3. 再启用新逻辑
4. 最后做 contract 清理

不要把这些步骤压成“一次发版全做完”。

## 10. 常见误区

### 10.1 “DOWN migration 写了就安全”

不对。

真正的生产安全来自：

- 前向兼容
- 演练
- 观测
- 灰度

不是来自一个理论上的回滚脚本。

### 10.2 “小改动不用走迁移流程”

不对。

很多大事故就是从“只是多加一列”开始的。

### 10.3 “数据库迁移只属于 DBA”

也不对。

业务开发必须理解：

- 结构如何演化
- 新旧代码怎么共存
- 数据如何回填

否则系统一到生产就会断层。

## 11. 失败模式

### 11.1 锁表事故

典型原因：

- 大表直接改高风险结构
- 索引创建方式不当
- 长事务和回填混在一起

### 11.2 新旧版本不兼容

典型原因：

- 新代码要求新列必有值
- 旧代码看不懂新语义
- 过早删除旧结构

### 11.3 回填压垮主库

典型原因：

- 一次性全量更新
- 没有限速
- 没有高峰规避

### 11.4 代码回滚成功，数据语义却没回去

这类问题最危险，因为表面上服务恢复了，实际数据已经进入半错状态。

## 12. 最后给一个落地建议

如果你在做一个新的 `.NET` 业务系统，更稳的默认路线通常是：

1. 所有数据库变更都进 migration
2. 所有高风险变更都按 `expand -> migrate -> contract` 设计
3. 数据回填和结构变更分开
4. 应用发布默认支持新旧版本共存
5. 高风险逻辑切换用 feature flag 灰度

这条路线的本质是：

> 把数据库变更当成生产交付问题，而不是脚本问题。
