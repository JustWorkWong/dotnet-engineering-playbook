# 模块化单体 vs 微服务

## 1. 先讲结论

对大多数 `.NET` 业务团队，更稳的默认答案不是“一上来微服务”，而是：

> 先做分层清晰、边界明确的模块化单体；只有当组织、部署、扩展和隔离收益已经明显大于治理成本时，再拆成微服务。

真正该比较的，不是“架构名字谁高级”，而是：

1. 你的团队规模和协作模式是什么
2. 领域边界是否已经稳定
3. 部署节奏是否真的冲突
4. 你是否已经具备分布式治理能力

## 2. 这场决策到底在解决什么问题

系统架构演化，本质上在解决四类矛盾：

1. 代码规模变大后，如何保持边界清晰
2. 团队协作变复杂后，如何降低互相踩踏
3. 上线节奏和扩容需求不同后，如何减少耦合
4. 故障、数据一致性和可观测性变复杂后，如何控制风险

很多团队过早上微服务，并不是这些问题真的成立，而是把“未来可能复杂”当成了“今天必须复杂”。

## 3. 什么是模块化单体

模块化单体不是“把所有代码放一个项目里”，而是：

- 单一部署单元
- 单一进程边界
- 模块内高内聚
- 模块间通过明确接口协作
- 数据访问和依赖方向有约束

它要解决的是：

- 保留单体开发和部署效率
- 提前训练边界意识
- 为未来拆分保留演化空间

一个更健康的模块化单体通常会有：

- 清晰的模块目录
- 明确的应用层和领域层边界
- 模块间不直接穿透数据库表
- 横切能力统一治理

## 4. 什么是微服务

微服务不是“项目多一点”，而是：

- 多个独立部署单元
- 独立运行进程
- 跨进程通信
- 独立扩容、独立发布、独立故障域
- 通常伴随独立数据边界和更强的平台治理

它真正解决的是：

- 团队自治
- 部署解耦
- 故障隔离
- 异构技术或资源需求分化

它带来的代价也非常真实：

- 网络不可靠
- 调用链更长
- 数据一致性更难
- 观测、部署、治理成本明显提高

## 5. 为什么模块化单体通常该先上

### 5.1 更适合边界仍在变化的业务

很多系统在前两年最难的不是“扛不住流量”，而是：

- 业务边界一直在变
- 规则频繁重写
- 团队还在摸真实聚合
- 数据关系还没稳定

这时如果过早拆服务，常见结果是：

- 服务边界按想象划，不按真实业务划
- 大量同步调用和共享数据库出现
- 最后只是把单体复杂度搬成分布式复杂度

### 5.2 更适合团队还没形成平台能力时

微服务要稳定运行，不只需要开发能力，还需要：

- CI/CD
- 配置中心
- 服务发现
- 可观测性
- 限流、重试、熔断
- 链路追踪
- 分布式事务和补偿

如果这些没准备好，微服务只会放大问题。

### 5.3 更适合先把业务做稳

模块化单体的核心价值是：

- 调试更直接
- 本地开发更简单
- 事务处理更自然
- 重构成本更低

在业务探索期，这些优势非常值钱。

## 6. 什么时候该认真考虑微服务

以下信号同时出现 3 个以上时，才更值得认真考虑拆分：

1. 不同模块的发布节奏长期冲突
2. 某些模块的资源需求和扩容策略完全不同
3. 团队已经按领域分化，且需要更高自治
4. 单体的改动范围和回归成本已经失控
5. 故障隔离需求很强，某个模块经常拖垮全局
6. 数据和业务边界已经相对稳定
7. 团队已经具备基本的平台治理能力

这里最关键的一条是：

> 不是系统大了就该拆，而是边界稳了、治理能力够了、收益清晰了才该拆。

## 7. 一个更实用的演化顺序

```text
简单单体
  -> 分层清晰的单体
  -> 模块化单体
  -> 先拆最独立的能力
  -> 逐步形成服务化边界
```

更稳的第一拆通常是这些类型：

- 异步处理模块
- 文件/媒体处理模块
- 报表/分析模块
- 高独立性的集成模块

不适合一开始就拆的，通常是：

- 强事务核心链路
- 边界还在变化的业务中心
- 高耦合共享数据模型

## 8. 两者最大的 trade-off

### 模块化单体的优势

- 开发效率高
- 本地调试简单
- 事务一致性更自然
- 重构成本低
- 对团队治理能力要求更低

### 模块化单体的代价

- 部署粒度较粗
- 扩容不能按模块细分
- 团队自治边界容易被突破
- 如果模块约束做不好，会退化成“大泥球单体”

### 微服务的优势

- 独立部署
- 独立扩容
- 故障隔离更强
- 团队按领域自治更自然

### 微服务的代价

- 分布式一致性复杂
- 网络和重试语义带来额外故障面
- 观测和平台成本更高
- 测试、联调、回归和排障显著变难

## 9. 真正的比较点，不在“代码怎么放”，而在“问题怎么变”

### 9.1 一致性问题

在单体里，很多问题可以靠本地事务解决。

在微服务里，同一个流程一旦跨服务：

- 就要面对最终一致性
- 就要设计 Outbox / Inbox
- 就要考虑补偿、幂等、重试和顺序性

所以很多团队拆完之后，第一反应不是“更优雅”，而是“为什么突然这么多 CAP、Saga、消息重放问题”。

### 9.2 故障问题

单体里的 bug 多半是进程内 bug。

微服务里的 bug 会升级成：

- 网络超时
- 上下游依赖雪崩
- 配置错配
- 链路不可见
- 局部失败导致全局状态不一致

### 9.3 组织问题

如果团队本身没有按领域拆清，微服务常会变成：

- 服务是拆了
- 团队还是互相卡
- 上下游沟通成本更高
- 发布协调更多

也就是说，微服务并不会替你完成组织设计，它只是把组织问题显性化。

## 10. 当前更稳的 `.NET` 实践建议

### 10.1 起步期

- 从模块化单体开始
- 在代码层就切清模块边界
- 数据访问按模块收口
- 日志、异常、配置、权限、审计统一治理

### 10.2 演化期

- 先补观测能力
- 先补异步能力和可靠消息
- 先补发布和回滚能力
- 再考虑服务拆分

### 10.3 服务化期

- 先拆独立性高的模块
- 服务间优先异步解耦
- 强事务链路谨慎拆
- 每拆一个服务，都要同步补告警、Trace、容量和恢复方案

## 11. 常见失败模式

### 11.1 假微服务

表面拆成多个服务，实际表现是：

- 共享数据库
- 大量同步强耦合调用
- 任何改动都要全链路联调

这不是服务化成功，而是把单体问题远程化了。

### 11.2 超前拆分

典型表现：

- 业务边界没稳就先拆
- 服务数量先涨，交付速度先掉
- 团队时间都花在网关、配置、超时、链路和环境问题上

### 11.3 单体不模块化

有些团队反过来踩坑：坚持单体，但没有模块边界，最后变成无法重构的大泥球。

这说明真正关键的不是“拆不拆”，而是：

> 无论是否跨进程，边界都必须先在代码和数据上成立。

## 12. 一个务实判断表

优先模块化单体，当：

- 团队规模中小
- 业务边界未稳定
- 本地事务很多
- 平台治理能力一般
- 交付速度是第一诉求

认真考虑微服务，当：

- 领域边界稳定
- 团队自治需求强
- 扩容和部署节奏明显分化
- 观测、CI/CD、消息治理已经具备

## 13. 最后给一个落地建议

如果你现在要做一个新的 `.NET` 业务系统，更稳的顺序通常是：

1. 先做模块化单体
2. 在单体阶段就把模块边界、数据归属和应用服务边界写清楚
3. 当某些模块的发布、扩容、故障域和团队归属真正独立后，再拆成微服务
4. 拆服务时优先拆高独立模块，不要先拆最核心强事务链路

这条路线的本质不是保守，而是：

> 先在进程内把边界练对，再把边界搬到网络上。
