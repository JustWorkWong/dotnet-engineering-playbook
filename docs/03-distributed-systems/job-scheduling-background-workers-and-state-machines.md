# 任务调度 / 后台作业 / 工作流状态机

## 1. 先讲结论

这三者不是一回事。

更稳的默认分层通常是：

- `任务调度`：回答“什么时候触发”
- `后台作业`：回答“谁来执行”
- `工作流状态机`：回答“这件事现在进行到哪一步”

很多系统一开始把所有事情都塞进 `BackgroundService`，结果很快就会遇到：

- 定时任务重入
- 作业重复执行
- 跨重启状态丢失
- 审批链和补偿路径讲不清

所以真正该问的不是“要不要开个后台线程”，而是：

1. 这是一次性作业、周期作业，还是长流程
2. 它需要跨重启恢复吗
3. 它有明确状态和人工节点吗

## 2. 先把三个概念分开

### 2.1 任务调度

它解决的是：

- 什么时候运行
- 间隔多久运行
- 是 cron、固定间隔，还是事件触发

调度本身不负责业务状态正确性。

### 2.2 后台作业

它解决的是：

- 谁来执行
- 执行失败怎么办
- 是否需要并发控制
- 是否需要重试

后台作业本质上是“执行面”。

### 2.3 工作流状态机

它解决的是：

- 当前状态是什么
- 下一步能走到哪里
- 哪些转换合法
- 超时、补偿、人工审核怎么表达

状态机本质上是“业务语义面”。

## 3. 一个更稳的默认判断

### 3.1 只需要“定时做点事”

比如：

- 清理过期数据
- 定时同步字典
- 刷新缓存

这类问题通常用：

- `BackgroundService`
- `PeriodicTimer`
- 调度器

就够了。

### 3.2 只需要“异步跑一段任务”

比如：

- 发送通知
- 导出报表
- 回填数据
- 生成文件

这类问题重点是：

- 队列
- 幂等
- 重试
- 反压

### 3.3 需要“跨多个节点推进，并且状态必须可解释”

比如：

- 审批流
- 订单履约
- AI 生成 -> 人工审核 -> 发布
- 补偿和回退链

这时就不该只靠后台作业循环了，而要上：

- 显式状态机
- 工作流编排
- 事件驱动状态推进

## 4. 为什么很多系统会设计错

因为它们把不同层的问题混在一起了：

- 调度器被迫保存复杂业务状态
- 后台作业被迫负责长流程语义
- 状态机又没有真正落到可恢复存储

结果就是：

- 代码里到处 `if status == ...`
- 一重启就不知道执行到哪
- 人工节点只能靠口头约定

## 5. 后台作业的更稳默认解

### 5.1 适合场景

- 处理队列消息
- 消费导出任务
- 执行回填任务
- 周期同步任务

### 5.2 当前更稳的做法

- 任务入队和执行分离
- 有界队列或显式调度
- 幂等执行
- 显式取消
- 有失败和重试策略

### 5.3 不该让它承担什么

- 复杂审批状态
- 跨多天多节点的流程
- 强审计且必须可回放的业务状态机

## 6. 调度器的更稳默认解

### 6.1 调度器真正负责什么

- 时间表达
- 触发规则
- 并发窗口
- 错过执行后的补跑策略

### 6.2 常见调度模式

- 固定间隔
- cron
- 延迟执行
- 事件触发后补定时检查

### 6.3 当前更稳的做法

- 调度器只负责触发，不负责复杂业务语义
- 调度任务必须可重入安全，或明确防重入
- 高风险任务不要靠内存 timer 存唯一事实

## 7. 状态机的更稳默认解

### 7.1 什么时候该上状态机

当你出现下面这些信号时：

1. 状态超过 3 到 5 个
2. 需要超时和补偿
3. 需要人工审核
4. 需要跨重启继续
5. 需要回放和审计

### 7.2 状态机真正的价值

不是“图更好看”，而是：

- 让合法状态转换显式化
- 让异常路径显式化
- 让恢复和人工介入有明确入口

### 7.3 当前更稳的做法

- 状态落库
- 转换事件结构化
- 超时和补偿有单独状态
- 人工节点有明确 ownership

## 8. 一个更实用的分层模型

```text
Scheduler
  -> enqueue job / trigger workflow step

Worker
  -> execute one bounded task

State Machine / Workflow
  -> decide current state and next legal transitions
```

这条边界很重要。

因为它能防止：

- 调度层侵入业务状态
- 执行层承担业务编排
- 流程层变成一堆隐式 if/else

## 9. `.NET` 里的更稳落地方式

### 9.1 简单周期任务

适合：

- `BackgroundService`
- `PeriodicTimer`

前提：

- 任务轻量
- 可丢一次问题不大
- 不需要复杂状态恢复

### 9.2 队列作业

适合：

- `Channel<T>`
- 消息队列消费者
- 独立 worker 服务

前提：

- 有幂等
- 有反压
- 有可观测性

### 9.3 长流程

适合：

- 显式 workflow 表
- 状态机字段
- 事件驱动推进
- 审计记录

必要时可以结合：

- CAP
- 消息队列
- AI workflow / MAF / orchestration

但重点仍然不是框架，而是状态是否清楚。

## 10. 常见误区

### 10.1 “BackgroundService 就是任务平台”

不对。

它更像宿主内执行容器，不天然提供：

- 持久状态
- 补偿
- 审批
- 回放

### 10.2 “有定时器就有调度系统”

不对。

定时器只能说明“会触发”，不能说明：

- 执行是否成功
- 是否重复
- 是否需要补跑

### 10.3 “状态写在代码分支里就算状态机”

不对。

如果状态转换没有显式建模，最终只会变成难维护的分支泥团。

## 11. 失败模式

### 调度失败模式

- timer 重入
- 多实例重复触发
- 节点重启后任务丢失

### 作业失败模式

- 队列无反压
- 重试导致副作用重复
- 失败后只有日志，没有恢复入口

### 状态机失败模式

- 状态散在多表多代码分支
- 审批节点没有超时策略
- 补偿路径靠人工临时决定

## 12. 最后给一个落地建议

如果你在做一个新的 `.NET` 业务系统，更稳的默认路线通常是：

1. 简单周期任务先用 `BackgroundService`
2. 中等复杂异步任务先用队列 + worker
3. 一旦出现长流程、审批、补偿和回放需求，就提升成显式状态机或 workflow

这条路线的本质是：

> 先把“触发、执行、状态”三层分开，再决定每层用什么技术。
