# 缓存 / 限流 / 重试 / 熔断

## 1. 先讲结论

这四个东西不是独立技巧，而是一组故障治理链：

- `缓存`：减少不必要的下游访问
- `限流`：限制系统在高峰和异常时的进入速度
- `重试`：处理短暂失败
- `熔断`：阻止失败放大

更稳的默认原则通常是：

> 先减少请求，再控制入口，再谨慎重试，最后在持续失败时快速熔断。

如果顺序反了，就很容易出现典型事故：

- 缓存没做好，数据库被打穿
- 限流没做好，流量尖峰直接压爆系统
- 重试乱开，把短故障放大成雪崩
- 熔断没做，下游挂了还持续打

## 2. 这组能力到底在解决什么问题

它们共同解决的是三类问题：

1. 容量问题
2. 瞬时故障问题
3. 故障扩散问题

很多团队把它们当成“性能优化”，但更本质的定位应该是：

- 缓存是容量治理
- 限流是入口治理
- 重试是短故障治理
- 熔断是故障隔离治理

## 3. 四者分别负责什么

### 3.1 缓存

缓存最适合解决：

- 热点只读数据
- 重复查询
- 下游昂贵调用
- 会话或短期状态

它最有价值的地方不是“更快”，而是：

- 降低数据库压力
- 平滑热点访问
- 给下游留下恢复空间

### 3.2 限流

限流最适合解决：

- 流量突增
- 恶意请求
- 下游容量有限
- 某类高成本操作需要配额控制

它不是“用户体验差”的代名词，而是保护系统和关键路径的手段。

### 3.3 重试

重试最适合解决：

- 网络瞬断
- 短时超时
- 下游暂时不可用
- 短暂锁冲突或资源竞争

但它只能处理短暂问题，不能替代业务正确性设计。

### 3.4 熔断

熔断最适合解决：

- 下游持续失败
- 重试开始放大故障
- 某个依赖已经不值得继续尝试

它本质上是在说：

> 现在继续打下去只会更糟，那就先断开，保护上游和整体系统。

## 4. 更稳的默认顺序

```text
缓存
  -> 限流
  -> 超时
  -> 重试
  -> 熔断
  -> 降级 / 回退
```

这条顺序很重要。

因为系统治理的逻辑应该是：

1. 先让请求尽量不要打到脆弱下游
2. 再限制进入系统的速度
3. 再给下游明确时间预算
4. 再对短暂失败做有限重试
5. 如果失败持续存在，立即熔断
6. 最后走降级或人工兜底

## 5. 缓存的更稳默认解

### 5.1 适合缓存什么

- 热点读
- 可容忍短暂陈旧的数据
- 组装成本高但更新频率不高的数据

### 5.2 不适合缓存什么

- 强一致写路径核心状态
- 极易变化且容错极低的数据
- 权限边界模糊的数据

### 5.3 当前更稳的做法

- 先定义 TTL，再讨论命中率
- 先定义回源策略，再讨论缓存粒度
- 先观测热 key / 大 key，再扩展缓存规模

## 6. 限流的更稳默认解

### 6.1 限流应该放在哪里

- API 网关
- 应用入口
- 高成本业务动作
- 下游依赖调用前

### 6.2 常见限流目标

- 每用户
- 每租户
- 每接口
- 每 IP
- 每类高成本操作

### 6.3 当前更稳的做法

- 关键接口默认有配额概念
- 限流结果要可观测、可告警
- 限流不是一刀切拒绝，可以按优先级降级

## 7. 重试的更稳默认解

### 7.1 什么时候可以重试

- 操作幂等
- 失败是短暂的
- 重试成本可控
- 有明确上限和退避策略

### 7.2 什么时候不要重试

- 非幂等写操作
- 下游已经持续不可用
- 当前失败其实是参数错误或权限错误

### 7.3 当前更稳的做法

- 重试必须有限次
- 默认指数退避
- 重试前先定义幂等边界
- 把超时、取消和重试一起设计

## 8. 熔断的更稳默认解

### 8.1 熔断到底保护谁

- 保护上游线程、连接和资源
- 保护下游不被雪上加霜
- 保护整体系统不被单点依赖拖垮

### 8.2 什么时候该熔断

- 失败率持续升高
- 平均延迟和超时持续恶化
- 重试开始堆积
- 下游已经进入故障状态

### 8.3 当前更稳的做法

- 熔断后必须有恢复探测
- 熔断状态必须能观测
- 熔断通常要配合降级结果，而不是简单抛异常

## 9. 这四者怎么配合

### 9.1 缓存和限流

缓存让大量重复请求不必回源。

限流让即使缓存失效或未命中，系统也不会瞬间被流量冲爆。

### 9.2 限流和重试

重试如果不受限流约束，很容易把下游的短故障放大。

所以真正成熟的系统里，重试不是“无限善意”，而是在配额和时间预算内受控执行。

### 9.3 重试和熔断

重试负责“再试一次也许能成”。

熔断负责“别试了，再试只会更糟”。

两者必须一起设计，否则系统会在两种极端间来回摆动：

- 要么完全不重试，过于脆弱
- 要么疯狂重试，把系统打崩

## 10. .NET 里的更稳落地方式

### 10.1 缓存

- `MemoryCache`：单机热点
- `Redis`：多实例共享缓存和短期状态

### 10.2 限流

- `ASP.NET Core Rate Limiting`
- 网关级限流
- 租户和用户维度配额

### 10.3 重试与熔断

- `Polly`
- `HttpClientFactory`
- 明确 timeout、retry、circuit breaker 和 fallback 组合

### 10.4 观测

必须补：

- 命中率
- 限流命中次数
- 重试次数
- 熔断状态
- 回退路径触发次数

没有这些指标，这四个能力就很容易变成“配了，但不知道效果和副作用”。

## 11. 常见误区

### 11.1 “命中率越高越好”

不一定。

如果为了命中率把 TTL 拉得很长，结果把脏数据风险放大，那就不是收益。

### 11.2 “失败就重试”

不对。

很多失败根本不该重试，比如：

- 参数错
- 权限错
- 业务规则不满足

### 11.3 “熔断就是报错”

不对。

真正成熟的熔断通常还会配：

- 回退结果
- 默认值
- 只读降级
- 人工兜底

## 12. 失败模式

### 缓存失败模式

- 缓存穿透
- 缓存击穿
- 缓存雪崩
- 热 key / 大 key

### 限流失败模式

- 限流粒度过粗
- 把正常核心流量一起误伤
- 限流结果不可观测

### 重试失败模式

- 非幂等操作被重复执行
- 下游越慢，上游重试越多
- 超时和重试叠加导致线程耗尽

### 熔断失败模式

- 熔断阈值不合理
- 熔断后无恢复探测
- 熔断只阻断了错误，没有准备降级路径

## 13. 最后给一个落地建议

如果你在做一个新的 `.NET` 业务系统，更稳的默认路线通常是：

1. 先定义缓存边界和 TTL
2. 再定义入口限流和高成本操作配额
3. 再给关键下游补 timeout + retry
4. 最后给关键依赖补熔断和降级

这条路线的本质不是堆中间件，而是：

> 把故障当常态治理，而不是等故障来了再写补丁。
