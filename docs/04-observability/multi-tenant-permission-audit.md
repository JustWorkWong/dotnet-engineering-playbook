# 多租户 / 权限 / 审计 生产设计

## 1. 先讲结论

多租户、权限、审计不该分开设计，它们本质上是一组业务治理边界：

- `多租户`：回答“这份数据属于谁”
- `权限`：回答“谁能做什么”
- `审计`：回答“谁在什么时候做了什么”

一个更稳的默认原则通常是：

> 先确定租户边界，再确定授权模型，最后把关键动作完整审计下来。

如果顺序反了，常见结果就是：

- 数据隔离不清
- 权限判断飘在接口层
- 出事后找不到是谁操作的

## 2. 这组设计到底在解决什么问题

它们共同解决的是三类核心问题：

1. 隔离
2. 控制
3. 追责

如果系统要进入真实生产，尤其是 B2B、SaaS、后台管理系统或 AI 业务平台，这三件事迟早都会变成主线。

很多团队一开始只关注“功能能不能做”，结果系统上线后最先追着补的往往正是这三类治理能力。

## 3. 三者分别负责什么

### 3.1 多租户

它负责定义：

- 数据归属
- 资源归属
- 配额归属
- 配置归属

最重要的问题不是“系统支不支持租户字段”，而是：

- 租户边界在哪
- 租户上下文怎么传播
- 租户隔离是逻辑隔离、库隔离还是实例隔离

### 3.2 权限

它负责定义：

- 谁能看
- 谁能改
- 谁能审批
- 谁能调用高风险能力

真正成熟的权限设计，不只是接口上挂一个 `[Authorize]`，而是要回答：

- 身份怎么确认
- 权限怎么建模
- 资源级判断怎么做
- 租户边界和权限边界如何叠加

### 3.3 审计

它负责回答：

- 谁
- 在什么时间
- 对哪个租户
- 做了什么动作
- 操作前后状态是什么

审计不是普通日志的别名。

普通日志更偏排障，审计更偏治理、合规、追责和业务复盘。

## 4. 更稳的默认设计顺序

```text
身份
  -> 租户上下文
  -> 权限判断
  -> 审计落库
```

这条顺序背后的逻辑是：

1. 先确认是谁
2. 再确认他属于哪个租户
3. 再判断他在当前租户里有没有权限
4. 最后把关键动作写入审计链

## 5. 多租户的更稳默认解

### 5.1 常见隔离模式

#### 逻辑隔离

- 同库同表
- 通过 `TenantId` 区分

适合：

- 中小规模 SaaS
- 成本敏感
- 运维复杂度要低

代价：

- 查询和过滤约束必须非常严格
- 一旦漏过滤，就容易越权读写

#### 数据库隔离

- 每租户独立库

适合：

- 大客户隔离要求高
- 数据迁移、备份、恢复需要更清晰边界

代价：

- 运维成本和连接管理成本更高

### 5.2 当前更稳的默认选择

多数业务系统的第一版，更稳的默认解通常是：

- 逻辑隔离起步
- 关键查询强制带 `TenantId`
- 后续对大客户再演进到库级隔离

### 5.3 必须同步治理的内容

- 租户上下文传播
- 配额
- 配置
- 缓存 key 设计
- 审计记录中的租户字段

## 6. 权限的更稳默认解

### 6.1 先区分认证和授权

- 认证：你是谁
- 授权：你能做什么

### 6.2 常见权限模型

#### RBAC

- 用户 -> 角色 -> 权限

适合：

- 后台管理系统
- 大多数标准业务系统

优点：

- 简单
- 易解释
- 易运维

#### ABAC

- 结合属性做细粒度判断

适合：

- 资源级控制复杂
- 条件很多
- 角色无法覆盖全部规则

代价：

- 更复杂
- 排障更难

### 6.3 当前更稳的默认路线

对大多数 `.NET` 业务系统，更稳的路线通常是：

```text
RBAC 起步
  -> 对少数高复杂资源补 ABAC 规则
```

不要一开始就把所有权限问题都做成通用策略引擎。

## 7. 审计的更稳默认解

### 7.1 哪些动作必须审计

- 登录、登出、权限失败
- 用户管理、角色变更、配置变更
- 审核、发布、删除、导出
- AI 高风险操作
- 工具调用和人工介入节点

### 7.2 一条审计记录至少包含什么

- `AuditId`
- `TenantId`
- `UserId`
- `Action`
- `ResourceType`
- `ResourceId`
- `OccurredAt`
- `Result`
- `TraceId`
- `Before` / `After` 摘要

### 7.3 当前更稳的默认做法

- 审计单独落库或单独表
- 关键字段结构化，不只存文本
- 审计和 Trace / Event 能互相回查

## 8. .NET 里的更稳落地方式

### 8.1 API / Middleware 层

负责：

- 身份解析
- 租户上下文提取
- `ClaimsPrincipal`
- 请求级审计上下文初始化

### 8.2 Application Service 层

负责：

- 真正的业务权限判断
- 资源级授权
- 关键动作触发审计事件

### 8.3 Infra 层

负责：

- 审计落库
- 租户过滤器
- 权限缓存
- TraceId / CorrelationId 关联

### 8.4 异步链路

如果进入消息、工作流或后台任务，必须继续传递：

- `TenantId`
- `UserId` 或操作主体
- `TraceId`

否则异步节点就会失去治理上下文。

## 9. 常见误区

### 9.1 “有 TenantId 字段就算多租户”

不对。

如果查询、缓存、消息、审计、导出链路没有一起带上租户边界，字段本身没有意义。

### 9.2 “有 [Authorize] 就算权限做好了”

不对。

接口鉴权只是入口，真正复杂的往往是：

- 资源级权限
- 审批权限
- 跨租户操作权限

### 9.3 “日志里能看到就算审计”

不对。

日志通常不够结构化，也不保证长期可追责。

## 10. 失败模式

### 多租户失败模式

- 漏带 `TenantId`
- 缓存 key 未分租户
- 异步消费丢失租户上下文

### 权限失败模式

- 只做角色判断，不做资源判断
- 高风险操作无二次校验
- 超级管理员逻辑到处散落

### 审计失败模式

- 只记录“成功”，不记录“失败尝试”
- 没有 before/after
- 审计和 Trace、业务记录断链

## 11. 最后给一个落地建议

如果你在做一个新的 `.NET` 业务系统，更稳的默认路线通常是：

1. 先确定租户隔离模式
2. 再用 `RBAC` 起步搭建权限
3. 对关键资源补资源级判断
4. 对关键动作补结构化审计
5. 让租户、权限、审计上下文沿同步和异步链路一起传播

这条路线的本质是：

> 不让治理能力漂在业务系统外面，而是把它做成业务边界的一部分。
