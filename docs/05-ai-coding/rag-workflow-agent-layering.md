# .NET 业务系统里的 RAG / Workflow / Agent 分层实践

## 1. 先讲结论

很多团队做 AI 系统时，最容易犯的错误不是模型不够强，而是分层错了。

更稳的默认顺序通常是：

```text
Prompt / Structured Output
  -> Tool Calling
  -> RAG
  -> Workflow
  -> Agent
```

不要反过来。

真正该问的不是“要不要上 Agent”，而是：

1. 这个问题缺的是知识、流程，还是自主决策
2. 哪一层已经够用
3. 再往上一层带来的收益，是否大于复杂度和风险

## 2. 这套分层到底在解决什么问题

把 AI 接入业务系统，本质上会遇到三种不同问题：

### 2.1 知识问题

模型不知道你的内部知识、最新规则、产品版本和私有文档。

这时重点是 `RAG`。

### 2.2 流程问题

系统不仅要回答，还要按步骤执行：

- 先校验输入
- 再检索资料
- 再调用工具
- 再进入审核
- 再写审计记录

这时重点是 `Workflow`。

### 2.3 自主决策问题

系统要在多个工具、多个步骤、多个失败恢复路径之间自己决定下一步。

这时才轮到 `Agent`。

很多团队一开始就上 Agent，本质上是在用最复杂的机制，去解决本来只需要知识层或流程层的问题。

## 3. 每一层分别负责什么

### 3.1 Prompt / Structured Output

适合：

- 分类
- 摘要
- 抽取
- 规则化输出

它解决的是“让模型按你要的格式说话”，不是“让模型接管系统”。

### 3.2 Tool Calling

适合：

- 查数据库
- 调外部 API
- 获取实时状态
- 执行业务动作前的受控查询

它解决的是“模型会说，但不会做”的问题。

### 3.3 RAG

适合：

- 企业知识问答
- 规范解释
- 产品规则检索
- 文档辅助决策

它解决的是“模型不知道你现在这份知识”的问题。

### 3.4 Workflow

适合：

- 多步骤固定流程
- 有审批和人工节点
- 有明确状态机
- 需要重试、补偿和可回放

它解决的是“步骤顺序和状态怎么被工程化控制”的问题。

### 3.5 Agent

适合：

- 步骤不完全固定
- 需要在多个工具间动态选择
- 需要根据中间结果调整策略
- 容忍一定非确定性

它解决的是“系统要不要拥有有限自主性”的问题。

## 4. 一个更稳的系统边界

在 `.NET` 业务系统里，更稳的分层通常是：

```text
API / App Service
  -> Workflow Orchestrator
  -> RAG Service / Tool Layer
  -> Model Gateway
  -> Audit / Metrics / Trace
```

如果再引入 Agent，它更适合站在：

```text
Workflow 内部的一个决策节点
或
受限工具集合上的自主执行器
```

而不是直接越过业务边界去“自由发挥”。

## 5. 为什么 RAG 和 Workflow 往往应该先于 Agent

### 5.1 RAG 先解决知识可信度

很多业务问题并不需要自主规划，而只是需要：

- 找到正确材料
- 带引用回答
- 基于当前规则解释

这时重点是知识层，而不是 Agent。

### 5.2 Workflow 先解决流程确定性

真实业务里，很多流程是固定的：

- 提交
- 校验
- 检索
- 分析
- 审核
- 落库

如果路径是固定的，就不该让 Agent 每次重新发明流程。

### 5.3 Agent 只接管真正不确定的部分

比如：

- 要从多个资料源里决定先查哪个
- 要根据中间失败结果选择替代工具
- 要决定是否继续深挖，还是请求人工介入

这时 Agent 才带来真实价值。

## 6. 在 `.NET` 里的落地方式

### 6.1 API 层

通常用 `ASP.NET Core` 暴露：

- 任务提交
- 状态查询
- 审核入口
- 审计查询

### 6.2 Workflow 层

负责：

- 固定流程编排
- 状态推进
- 超时、重试、补偿
- 人工节点和回放

这里可以用：

- 自己写显式状态机
- `BackgroundService + Queue`
- `CAP + 事件驱动编排`
- `MAF / SK` 中的流程编排能力

### 6.3 RAG 层

负责：

- 文档入库
- chunk 和 metadata
- 检索
- rerank
- citation

### 6.4 Agent 层

负责：

- 在受限工具集合中做动态选择
- 基于中间结果决定下一步
- 处理非固定子任务

注意：

> Agent 更像“受限决策器”，不是无边界业务核心。

### 6.5 观测层

必须统一补齐：

- trace
- metrics
- logs
- token/cost
- tool 调用记录
- 审计事件

否则你根本无法解释：这次失败是检索错了、流程卡了，还是 agent 决策偏了。

## 7. 一张更实用的选型表

| 场景 | 更适合的层 |
| --- | --- |
| 文档问答、规则解释 | RAG |
| 固定多步骤审批流程 | Workflow |
| 需要动态调用多个工具 | Agent |
| 简单抽取和结构化输出 | Prompt / Structured Output |
| 查实时库存、订单状态、执行查询 | Tool Calling |

## 8. 当前更稳的默认组合

对大多数企业 AI 系统，一个更稳的默认组合通常是：

```text
Structured Output
  + Tool Calling
  + RAG
  + Workflow
  + Human Review
```

只有当下面条件成立时，再考虑补 Agent：

1. 步骤确实不稳定
2. 工具选择确实需要动态决策
3. 团队能接受更高的调试和观测成本
4. 失败时有人工兜底和回退机制

## 9. 常见误区

### 9.1 把 Agent 当高级 Workflow

如果步骤本来就是固定的，Agent 只会增加不确定性。

### 9.2 把 RAG 当万能知识补丁

如果文档源混乱、版本脏、metadata 缺失，RAG 只会稳定地检索出错误材料。

### 9.3 把 Workflow 当代码堆叠

如果没有状态、超时、补偿和人工节点设计，所谓 Workflow 只是把步骤写长了。

### 9.4 把 Claude Code / Codex / ChatGPT 和运行时 Agent 混为一谈

这三者更偏开发阶段的生产力工具。

业务运行时的 Agent 是系统运行面问题，不是研发协作面问题。两者虽然都叫 agent，但约束和风险完全不同。

## 10. 失败模式

### RAG 失败模式

- 召回错文档
- 上下文噪声过大
- 引用存在但答案不 grounded

### Workflow 失败模式

- 状态机不清
- 重试把副作用放大
- 人工节点缺少超时和回收策略

### Agent 失败模式

- 工具选择漂移
- 调用过多导致成本爆炸
- 决策不可解释
- 失败恢复路径不稳定

## 11. 一个更务实的演化顺序

### 第一阶段

- 先做 `Prompt + Structured Output`
- 让结果可控、可解析、可验证

### 第二阶段

- 再补 `Tool Calling`
- 让模型能查询实时信息，而不是硬编

### 第三阶段

- 再补 `RAG`
- 让系统接入企业知识和引用能力

### 第四阶段

- 再补 `Workflow`
- 让步骤、状态、审核和恢复可控

### 第五阶段

- 最后才补 `Agent`
- 只让它接管真正不确定的子问题

## 12. 最后给一个落地建议

如果你现在在做一个 `.NET` AI 业务系统，更稳的方案通常不是“先上多 Agent”，而是：

1. 先把知识层和工具层做准
2. 再把流程层做稳
3. 最后只在必要位置引入 Agent

这条路线的本质是：

> 先用确定性控制系统，再把有限自主性交给那些真的需要自主性的地方。
