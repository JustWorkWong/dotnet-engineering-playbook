# AI 成本 / 评测 / 安全护栏

## 1. 先讲结论

AI 系统进生产后，最容易失控的三件事是：

- 成本
- 质量
- 风险

所以更稳的默认原则通常是：

> 成本要可预算，质量要可评测，风险要有护栏。

如果这三件事没补齐，系统很容易出现：

- token 和模型账单失控
- 演示效果很好，真实请求质量却不稳定
- 工具调用越权、泄露敏感信息、输出不可控

很多团队前期只关注“能不能跑通”，但真正进入业务后，核心问题会迅速变成：

1. 这次调用值不值得
2. 这次结果靠不靠谱
3. 这次失败会不会带来安全或业务事故

## 2. 先把三件事分开

### 2.1 成本

成本不只是 token 价格。

它至少包括：

- 模型调用费用
- embedding 和 rerank 成本
- 工具调用成本
- 工作流重试成本
- 人工审核成本
- 延迟带来的业务成本

### 2.2 评测

评测不只是“看起来回答不错”。

它更应该回答：

- 是否命中了正确知识
- 是否按预期使用工具
- 是否满足结构化输出要求
- 是否在真实场景里稳定

### 2.3 护栏

护栏不只是内容安全过滤。

它还包括：

- 输入边界
- 工具权限
- 数据脱敏
- 输出约束
- 人工审批
- 高风险动作阻断

## 3. 为什么这三者必须一起设计

因为它们彼此会直接影响：

- 没有评测，你就不知道成本优化是不是把质量优化没了
- 没有护栏，质量再高也可能因一次越权调用出事故
- 没有成本治理，再好的系统也很难长期上线

所以更稳的工程顺序不是：

```text
先把效果做出来
以后再管成本和风险
```

而应该是：

```text
定义目标
  -> 定义评测
  -> 定义预算
  -> 定义护栏
  -> 再放量
```

## 4. 成本的更稳默认解

### 4.1 先把成本拆账

至少按这些维度看：

- 每请求成本
- 每租户成本
- 每功能成本
- 每个 workflow 成本
- 每个工具调用成本

如果所有调用都混在总账单里，团队很快就会失去优化抓手。

### 4.2 当前更稳的默认做法

- 先设预算，再谈放量
- 默认优先小模型和短上下文
- 只有评测证明不够时，再升级模型或加上下文
- 记录 prompt token、completion token、tool 次数、延迟

### 4.3 什么时候成本最容易失控

- 上下文无限堆
- RAG 召回过多 chunk
- agent 不受控多轮调用
- 重试没有上限
- 大模型默认全流量直上

## 5. 评测的更稳默认解

### 5.1 先定义 eval，而不是先凭感觉调

更稳的默认做法通常是：

- 先写能力 eval
- 再写回归 eval
- 再做灰度放量

这和写软件测试的思路一样：

> 没有评测标准，就谈不上稳定迭代。

### 5.2 常见评测维度

- 正确率
- 引用是否 grounded
- 工具调用是否合理
- 输出格式是否稳定
- 审核通过率
- 人工返工率

### 5.3 RAG 场景重点看什么

- 召回质量
- 引用正确性
- 答案是否真的被证据支持

### 5.4 Workflow / Agent 场景重点看什么

- 是否走对流程
- 是否在正确节点停下
- 是否误调工具
- 是否成本可控

## 6. 护栏的更稳默认解

### 6.1 输入护栏

负责：

- schema 校验
- 长度限制
- 文件类型限制
- prompt injection 基础防护

### 6.2 工具护栏

负责：

- 工具白名单
- 参数校验
- 高风险工具审批
- 读写权限边界

### 6.3 数据护栏

负责：

- PII 脱敏
- 密钥和凭证保护
- 租户边界
- 审计留痕

### 6.4 输出护栏

负责：

- Structured Output
- 高风险内容拦截
- 置信度不足时降级
- 必要时只返回证据，不直接下结论

## 7. 一个更实用的 AI 生产链

```text
Input Validation
  -> Retrieval / Tool Access
  -> Model Call
  -> Output Validation
  -> Audit / Metrics / Cost Recording
  -> Human Review (if needed)
```

这条链里每一段都不是可选装饰。

因为 AI 系统真正危险的地方，不是模型“答错一句话”，而是：

- 错得很像对
- 工具调错了
- 泄露了不该看的数据
- 成本在静默中爆炸

## 8. `.NET` 里的更稳落地方式

### 8.1 成本观测

至少记录：

- `TenantId`
- `UserId`
- `Model`
- `PromptTokens`
- `CompletionTokens`
- `LatencyMs`
- `ToolCallCount`
- `EstimatedCost`
- `TraceId`

### 8.2 Eval 落地

更稳的做法通常是：

- 用固定样本集做 capability eval
- 用关键链路做 regression eval
- 新 prompt / 新模型 / 新 workflow 改动都跑一次

### 8.3 护栏落地

可以放在：

- API 层输入校验
- Tool 调用包装层
- 模型输出验证层
- 审计和审批层

关键不是放哪一层最优雅，而是：

> 高风险动作必须有显式阻断点。

## 9. 常见误区

### 9.1 “先把效果做出来，再补 eval”

这会导致后面每次优化都像撞运气。

### 9.2 “成本高就换更小模型”

不一定。

真正的问题也可能在：

- prompt 太长
- chunk 太多
- tool 调用失控
- workflow 回路太深

### 9.3 “安全护栏就是内容审核”

不对。

AI 系统更真实的风险往往是：

- 越权访问
- 敏感数据泄露
- 错误执行工具
- 审批绕过

## 10. 失败模式

### 成本失败模式

- 上下文持续膨胀
- 多 agent 互相放大调用
- 无预算上限

### 评测失败模式

- 只有 demo case，没有真实回归集
- 只看主观满意度，不看结构化指标
- 改模型后无基线对比

### 护栏失败模式

- 工具白名单缺失
- 输出未校验直接执行
- 高风险动作无人工节点
- 审计不完整，事后无法回放

## 11. 更稳的默认放量顺序

### 第一阶段

- 小流量
- 严护栏
- 全审计
- 强人工复核

### 第二阶段

- 增加 eval 集
- 优化成本
- 缩小人工复核范围

### 第三阶段

- 分租户配额
- 分风险等级护栏
- 更细的模型路由

## 12. 最后给一个落地建议

如果你在做一个新的 `.NET` AI 业务系统，更稳的默认路线通常是：

1. 先把成本记录全
2. 再把 eval 定义清楚
3. 再把工具和输出护栏补齐
4. 最后才扩大流量和自动化半径

这条路线的本质是：

> 让 AI 系统先可控，再可扩，再可放大。
